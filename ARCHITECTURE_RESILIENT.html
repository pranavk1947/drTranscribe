Engineering Principles Applied

1. Circuit Breaker Pattern

Each external provider call goes through a circuit breaker. After N consecutive failures, the circuit opens and traffic instantly routes to the next provider in the priority chain. After a cooldown period, the circuit enters half-open state and tests the original provider with a single request. This prevents cascading failures and eliminates provider downtime as a SPOF.

Resilience

2. Bulkhead Isolation

STT and LLM failures are isolated in separate circuit breakers. A Groq outage cannot affect GPT extraction. Each session is processed independently — one session's failure never impacts another. The pipeline stages (transcription, classification, extraction) are isolated so a classifier failure can be bypassed while STT + LLM continue working.

Resilience

3. Distributed State (Redis)

Replaces in-memory session dict with Redis Cluster. Sessions survive server restarts. Multiple FastAPI workers share state via Redis, enabling horizontal scaling. TTL-based cleanup prevents memory leaks. Pub/sub enables real-time session events across workers. Eliminates the single-process server as a SPOF.

No SPOF

4. Dead Letter Queue + Retry

Failed audio chunks are not silently dropped — they enter a DLQ. A retry worker processes them with exponential backoff (1s, 2s, 4s, 8s...) up to a max retry count. After exhausting retries, chunks are logged for manual inspection. The full transcript on next successful chunk recovers context from prior failures.

Fault Tolerance

5. Health Monitoring

Active health monitor tracks: provider response latency (P50/P95/P99), circuit breaker states (closed/open/half-open), error rates per provider, and queue depth. Exposes /health (liveness) and /readiness (can accept traffic) endpoints. Enables proactive alerting before failures cascade.

Observability

6. Graceful Degradation

The system degrades gracefully at every layer: if Classifier is down, extraction proceeds without enriched context. If all STT providers fail, chunks are queued for retry. If Redis is unreachable, fall back to local in-memory state with a warning. If WebSocket drops, the client preserves all extraction data captured so far.

Fault Tolerance

7. Provider Pool with Priority Chain

No single AI provider is a SPOF. STT has a 4-deep fallback chain (Groq → Gemini → OpenAI → Azure). LLM has a 2-deep chain (Azure GPT → OpenAI GPT). The circuit breaker automatically routes to the next healthy provider. Adding a new provider is a config change, not a code change (Strategy Pattern).

No SPOF

8. Classifier as Enrichment Layer

The Classifier sits between transcription and extraction. It detects medical intent (diagnosis, prescription, follow-up), extracts entities (drug names, symptoms, body parts), and determines speciality context. This enriched context improves extraction accuracy. The Classifier is an external service — if it fails, the pipeline continues with raw transcript (graceful degradation).

Separation of Concerns

Classifier Pipeline Detail

Extraction Pipeline — With Classifier Enrichment

Why a Classifier?

The current system sends raw transcript directly to the LLM for extraction. Adding a Classifier between transcription and extraction provides three advantages:

  1. Better extraction accuracy — The LLM receives pre-identified entities and intents, reducing hallucination and missed data
  2. Speciality-aware prompts — A cardiology consultation needs different extraction focus than an orthopaedic one. The Classifier detects speciality and the Extraction Service selects the optimal prompt template
  3. Cost optimization — Lightweight classification (fast, cheap model or rule-based) pre-filters noise before sending to the expensive LLM. Short or irrelevant chunks can be skipped entirely

Graceful Degradation

If the Classifier service is unavailable, the pipeline falls back to sending the raw transcript directly to the Extraction Service (current V2 behavior). The Classifier is an enrichment layer, not a hard dependency.
